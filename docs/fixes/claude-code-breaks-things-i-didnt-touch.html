Claude Code keeps breaking things I didn't touch

By Sandeep Roy · June 6, 2026 · 4 min read

You ask Claude Code to fix one small thing. It fixes it — and "helpfully" refactors a component you never mentioned, renames a prop three files away, and breaks two features that were working fine an hour ago. You didn't touch that code. Claude did.

This is the universal grievance. "Claude Code keeps breaking things I didn't touch" is literally the title of a widely-shared post, and the #1 vibe-coding complaint on Reddit is blunt: "it will fix one thing but destroy 10 other things in your code." The agents don't do what they're told — they do what they want.

Why scope creep happens

Putting "only change what I ask for" in your CLAUDE.md doesn't stop it, for the same reason every other rule gets ignored:

  • Rules are suggestions, not fences. CLAUDE.md is read as context. Nothing physically prevents the model from editing a file it was told to leave alone — there's no enforcement between "the rule exists" and "the file gets rewritten."
  • The instruction limit erodes guardrails. LLMs reliably follow only about 150–200 instructions. As your session grows and your rules file lengthens, "don't touch the auth files" competes with everything else and loses. By mid-session it's effectively gone.
  • "Helpful" is the default. The model is optimized to improve things. Absent a hard boundary, it treats nearby code as fair game and "improves" it into a broken state.

What you actually want is a fence around the code the AI must not touch — not a politely-worded request it forgets ten messages later.

The fix: lock the files, let it work everywhere else

SpecLock is an npm package and MCP server that reads your existing rules file (CLAUDE.md, .cursorrules, or AGENTS.md) and turns "never touch X" into a runtime hard constraint. Before any change is applied, SpecLock checks the patch's file list against your locks. An edit that strays into protected territory gets blocked — even though you asked for something else entirely.

Two lines:

cd your-project
npx speclock protect

npx speclock protect reads your rules, compiles each "don't touch" into a lock, and enforces it across Claude Code, Cursor, Windsurf, Cline, and Copilot. If you don't have a scope rule yet, add one line to your CLAUDE.md — NEVER modify files in /auth or /payments — and SpecLock locks it.

The whole idea: lock the files it keeps breaking. Let it work freely everywhere else. The AI stays useful where it's safe and physically can't reach where it isn't.

Before / after

You have a lock from your CLAUDE.md: "NEVER modify auth files." You ask: "Fix the typo in the signup button label."

Before SpecLock — Claude fixes the label, then "tidies up" auth/session.ts while it's in there and quietly breaks token refresh. You find out in production.

After SpecLock — the in-scope edit goes through; the out-of-scope one is stopped:

$ claude: fix the typo in the signup button label
✓ edit components/SignupButton.tsx — allowed
→ also proposes: refactor auth/session.ts

🔒 SpecLock BLOCKED this action
  Constraint: "NEVER modify auth files."
  Source:    CLAUDE.md
  Conflict:  HIGH — patch edits auth/session.ts (out of scope)
  Override with: speclock override —reason "..."

The typo gets fixed. The auth file doesn't get touched. The change you didn't ask for is now impossible by accident. Run speclock wins for a shareable "Save Receipt" of every break it prevented.

The honest version

SpecLock won't make Claude less eager to "help." It just draws a hard line the help can't cross. You keep the speed of an AI agent and lose the part where it silently wrecks code three directories away.
