CLAUDE.md not being followed

By Sandeep Roy · June 6, 2026 · 4 min read

You added a CLAUDE.md with clear, sensible rules. And Claude Code acts like half of them aren't there. Some get respected, some get ignored, and there's no obvious pattern — until you realize the pattern is the problem.

If you searched "CLAUDE.md not being followed," here's the short answer first: it's not a config bug, and writing the rules more emphatically won't fix it. The reason is structural, and so is the fix.

Why CLAUDE.md isn't being followed

Two independent forces are at play, both well documented (HN, setup guides):

  • No enforcement mechanism. CLAUDE.md is loaded as context. Claude reads your rules but nothing blocks a violation. It's the equivalent of rules posted on a bulletin board — seen, then ignored.
  • The ~150-instruction limit. LLMs reliably follow only about 150–200 distinct instructions, and Claude Code's own prompt consumes a chunk of that budget before your file loads. A long CLAUDE.md silently dilutes every rule; the ones near the bottom get dropped first, and a growing conversation pushes them all further from the model's attention.

So the behavior you see — top rules followed, bottom rules intermittent or ignored — is exactly what you'd predict. The file isn't broken. It just has no teeth.

The fix: give your CLAUDE.md teeth

SpecLock is an npm package and MCP server that reads the CLAUDE.md you already wrote (plus .cursorrules and AGENTS.md) and compiles each rule into a runtime hard constraint. Before any AI-generated change is applied, SpecLock checks it against your locked rules and blocks anything that would break one. Length and position stop mattering — enforcement doesn't depend on the model remembering.

Two lines, no rewrite:

cd your-project
npx speclock protect

npx speclock protect discovers your existing rules, turns each into a lock, and enforces them across Claude Code, Cursor, Windsurf, Cline, and Copilot. No new rule format, no dashboards, no hand-written hooks.

What changes: a rule on line 200 is enforced exactly as hard as a rule on line 2. "Followed sometimes" becomes "checked every time."

Before / after

Your CLAUDE.md says (near the bottom): "Never delete test files."

Before SpecLock — you ask Claude to "clean up unused code," it decides some tests cover "deprecated" features, and your coverage quietly drops from 85% to 40%.

After SpecLock — same request, intercepted:

$ claude: clean up unused code
→ proposes: delete tests/legacy/*.test.ts

🔒 SpecLock BLOCKED this action
  Constraint: "Never delete test files."
  Source:    CLAUDE.md
  Conflict:  HIGH — patch removes 6 test files
  Override with: speclock override —reason "..."

The deletion is now visible and deliberate instead of silent. You reject it or override it on purpose — and speclock wins gives you a shareable "Save Receipt" of everything it blocked.

The honest version

SpecLock doesn't raise the instruction limit or change how Claude reads your file. It adds the missing enforcement layer so your CLAUDE.md behaves like a contract instead of a wish list. That's the entire difference between "not being followed" and "can't be broken."
