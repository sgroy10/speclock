Claude ignores CLAUDE.md

By Sandeep Roy · June 6, 2026 · 4 min read

You wrote a careful CLAUDE.md. Stack, conventions, the hard "never do this" rules. For the first few prompts Claude respects them. Then the session gets longer, and suddenly it's doing exactly the thing you told it never to do — like it never read the file at all.

If you've searched "Claude ignores CLAUDE.md," you're in good company. It's now a textbook complaint on Hacker News and across the setup guides: rules at the top of the file get followed, rules near the bottom get followed intermittently or not at all.

Why this happens (it's not you, it's the instruction limit)

This isn't Claude being lazy. It's a measurable property of how large language models follow instructions. The consistent finding across practitioners: an LLM reliably follows only about 150–200 distinct instructions at once. Claude Code's own system prompt already eats roughly 50 of those before your file is even loaded.

So when your CLAUDE.md grows past a page or two, every additional rule competes for a shrinking pool of attention. The result:

  • Long files get diluted. The more rules you add, the weaker each one becomes. Bolding and ALL CAPS don't fix an attention problem.
  • Bottom-of-file rules get dropped first. Position matters — the rule you put at line 200 is the one most likely to be silently skipped.
  • Conversation length makes it worse. As the session fills the context window, CLAUDE.md drifts further from the model's focus. By message 15 it's effectively forgotten.

The root cause is structural: CLAUDE.md is a suggestion, not a contract. Claude reads it, but nothing stops it from violating a rule. There is no enforcement layer between "the rule exists" and "the action happens."

The fix: enforce the file you already wrote

SpecLock is an npm package and MCP server that reads your existing CLAUDE.md (and .cursorrules, and AGENTS.md) and turns those prose rules into runtime hard constraints. Instead of hoping Claude remembers a rule, SpecLock checks every proposed action against your locked rules and blocks the ones that would break them — no matter how long your file got or where the rule sits.

Setup is two lines. You don't rewrite anything:

cd your-project
npx speclock protect

That's it. npx speclock protect discovers your current rules file, compiles each rule into a lock, and starts enforcing across Claude Code, Cursor, Windsurf, Cline, and Copilot. No new rule syntax, no dashboard, no scripting.

What changes: the rule at the bottom of your CLAUDE.md gets enforced exactly as hard as the rule at the top. File length stops mattering, because enforcement no longer depends on the model paying attention.

Before / after

Say line 180 of your CLAUDE.md reads: "NEVER switch the framework. We use Express."

Before SpecLock — you ask Claude to "optimize the routing," and it cheerfully migrates you to Fastify. The rule was there. Nothing checked it.

After SpecLock — the same request, intercepted before the change lands:

$ claude: optimize the routing layer
→ proposes: replace express import with fastify

🔒 SpecLock BLOCKED this action
  Constraint: "NEVER switch the framework. We use Express."
  Source:    CLAUDE.md (line 180)
  Conflict:  HIGH — patch introduces a Fastify import
  Override with: speclock override —reason "..."

The violation is now visible and deliberate instead of silent and accidental. You either reject it or override it on purpose. And SpecLock keeps a receipt — run speclock wins to see a shareable "Save Receipt" of everything it blocked.

The honest version

SpecLock doesn't make Claude smarter or extend the instruction limit. It moves your rules out of the "please remember this" layer and into a "this is checked every time" layer. That's the whole trick — and it's why a long CLAUDE.md stops being a liability.
