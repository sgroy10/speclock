Cursor just ignores my rules

By Sandeep Roy · June 6, 2026 · 4 min read

You set up a .cursorrules file. You were specific. "Use Tailwind, not inline styles." "Don't touch the migrations folder." "Keep the API surface stable." And Cursor agreed — right up until it didn't, quietly steamrolling the exact rule you cared about most.

"Cursor just ignores my rules" is one of the most-upvoted threads on the Cursor forum, with a sibling thread titled ".cursorrules not being followed or applied." This is a known, structural problem — not a config mistake you made.

Why .cursorrules gets ignored

Two things are working against you, and neither is fixable by writing better rules:

  • There's no enforcement layer. .cursorrules is a static text file that gets fed to the model as context. The AI sees your rules but nothing blocks it from violating them. It's office rules taped to the wall — visible, ignorable.
  • The instruction limit dilutes long files. LLMs reliably follow only about 150–200 instructions at a time. The longer your rules file grows, the weaker each individual rule becomes, and the ones near the bottom get dropped first. As the chat fills the context window, your rules drift out of the model's attention entirely.

So the rule that "works on Monday" silently stops working by message twelve on Wednesday. Same file, same rule — just no mechanism keeping it honest.

The fix: enforce the .cursorrules you already wrote

SpecLock is an npm package and MCP server that reads your existing .cursorrules (and CLAUDE.md, and AGENTS.md) and compiles each rule into a runtime hard constraint. Before any Cursor-generated change is applied, SpecLock checks it against your locked rules and blocks the ones that would break them. Position in the file and length of the file no longer matter, because enforcement doesn't depend on the model remembering.

Two lines, no rewrite:

cd your-project
npx speclock protect

npx speclock protect discovers your current .cursorrules, turns each rule into a lock, and enforces it across Cursor, Claude Code, Windsurf, Cline, and Copilot via the MCP standard. No new syntax, no SaaS account, no hand-written hooks.

What changes: Cursor can still suggest anything — but a suggestion that violates a locked rule gets stopped before it touches your files, with a clear note about which rule it broke.

Before / after

Say your .cursorrules contains: "Never edit anything in /db/migrations without approval."

Before SpecLock — you ask Cursor to "clean up the schema," and it rewrites a migration in place. The rule existed. Nothing checked it.

After SpecLock — same request, intercepted:

$ cursor: clean up the schema
→ proposes: edit db/migrations/0007_users.sql

🔒 SpecLock BLOCKED this action
  Constraint: "Never edit /db/migrations without approval."
  Source:    .cursorrules
  Conflict:  HIGH — patch modifies a protected migration
  Override with: speclock override —reason "..."

The violation is now visible and intentional instead of silent. You reject it, or override it on purpose. SpecLock also keeps a receipt — speclock wins shows a shareable "Save Receipt" of everything it blocked.

The honest version

SpecLock doesn't patch Cursor or change how the model thinks. It adds the missing layer: a check that runs every time, so your .cursorrules behaves like a law instead of a polite request. That's the difference between rules Cursor reads and rules Cursor can't break.
