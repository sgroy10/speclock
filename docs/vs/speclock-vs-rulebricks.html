SPECLOCK vs RULEBRICKS GUARDRAILS

SpecLock vs Rulebricks Guardrails: your rules file, or a SaaS decision table?

Both enforce AI tool calls in real time. The difference is where the rules live. Rulebricks runs them from a separate hosted decision table, Claude-Code-only. SpecLock reads the rules file you already wrote — self-contained and cross-agent.

rulebricks/claude-code-guardrails (~70 stars) is a real enforcement tool, and credit where it's due: it actually blocks. It intercepts Claude Code tool calls in real time and evaluates them against rules — so unlike pure sync tools, a violation gets stopped, not just discouraged.

The trade-off is where those rules come from. Rulebricks evaluates against decision tables you build in the Rulebricks platform — a separate SaaS product. The enforcement logic isn't derived from your CLAUDE.md; it lives in a hosted table you author and maintain elsewhere, and it requires a Rulebricks account. It's also Claude Code only — the guardrails don't extend to Cursor, Windsurf, Cline, or Copilot.

SpecLock takes the opposite approach. There's no second platform and no decision table to build. It reads the CLAUDE.md / .cursorrules / AGENTS.md you already wrote, compiles those prose rules into hard runtime locks, and enforces them across every major agent — fully self-contained, free, and open source.

When to use which

Use Rulebricks Guardrails if…

You're already a Rulebricks user, you're fully on Claude Code, and you want enforcement logic expressed as formal decision tables — spreadsheets of conditions and outcomes managed in a dedicated rules engine. If your organization already standardizes on Rulebricks for business rules, reusing that platform for AI guardrails is a coherent choice.

Use SpecLock if…

You don't want a second platform in the loop, and you'd rather the rules you already wrote in your repo be the source of enforcement — no decision table to build or keep in sync. Especially if you use more than one AI agent, SpecLock gives you self-contained, cross-agent enforcement straight from your CLAUDE.md, plus drift scoring and a shareable receipt of what it blocked.

The honest summary

Both genuinely enforce. Rulebricks centralizes rule logic in an external decision engine for Claude Code; SpecLock keeps everything in your repo and works across agents. If your rules already live in CLAUDE.md and you want zero extra infrastructure, SpecLock is the lower-friction path.
